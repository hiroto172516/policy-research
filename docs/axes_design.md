# 評価軸の設計（STEP2）― ヒアリング結果メモ準拠

> 出典: `../../ヒアリング結果メモ.md`（現時点で唯一の一次情報）。本設計は**このメモだけ**を根拠にし、
> 外部フレームワークを持ち込まない。実体は `config/axes.yml`（v0.2）。
> 位置づけ: **JCLP（武井氏）レビューで確定させる叩き台**。「評価軸の設計はJCLPの役割」（memo L54-59）。

## 1. 全体像 ― 議員を4つの軸で捉える

メモ L24「脱炭素…への**関心・スタンス**を分類」「**類型化**」、L9「立場・**影響力**」を素直に分解すると、
議員は次の4軸で表現できる。

| # | 軸 | 何を測るか | データの由来 | memo根拠 |
|---|----|-----------|-------------|---------|
| ① | トピック別**スタンス** | 各政策テーマへの賛否の向き（5段階） | **発言**（STEP3で分類） | L24 |
| ② | **関心度**（engagement） | どれだけ言及しているか（高/中/低/なし） | **発言** | L24,L65 |
| ③ | **類型**（archetype） | ①②を束ねた議員タイプ | **発言**（①②から導出） | L25 |
| ④ | **影響力**（Tier）＋seniority | キーマンか（役職）／当選歴 | **名簿**（roster・発言と独立） | L9,L26-28 |

①②③は「発言という事実」からAIが分類する部分（STEP3）。④は名簿から機械的に算出する部分で、
**発言収集(403で保留中)を待たずに今すぐ算出できる**（`compute_influence.py`）。

## 2. ①トピック（評価軸）とメモ対応

メモ L24 が明示する5テーマ＋ L92-93 の「時期により変化する関連トピック」を採用。

| topic | group | memo | 備考 |
|-------|-------|------|------|
| 脱炭素・気候変動 | core | L24,L92 | 大枠テーマ（不変） |
| 再生可能エネルギー | core | L24,L25 | 類型「再エネ積極支持」の基礎 |
| 太陽光発電 | dynamic | L24,L25,L93 | 「太陽光批判」の基礎。景観・農地・安保 |
| 原子力 | dynamic | L93 | 時期で重要度が変化 |
| 電力需要（AI・データセンター） | dynamic | L93 | 需要増・電源確保 |
| EV・電動化 | core | L92 | 大枠テーマ |
| 経済安全保障 | core | L24,L25 | 「経済安保重視」の基礎 |
| エネルギー政策全般 | core | L24 | 電源構成・電気料金 |

- **core=大枠で不変／dynamic=時期・社会トレンドで動く**（memo L90-94）。dynamic の顔ぶれは
  収集の観点とともに随時見直す（＝`axes.yml` を差し替えるだけ）。

## 3. ②関心度・③類型

- **関心度**（高/中/低/なし）: スタンス（賛否の向き）とは別軸。**発言が少ない議員も落とさない**ため
  「なし」を明示的に持ち、類型 `low_engagement` として監視対象に残す（memo L65）。
- **類型**: メモ L25 の3例をそのまま初期類型に採用（再エネ積極支持／太陽光批判／脱炭素関心薄・経済安保重視）。
  dynamic を反映して `原子力・安定供給重視` を補い、`発言少・様子見`／`分類保留` を加える。

## 4. ④影響力（Tier）― 実データ算出済み

memo L9「影響力」・L26「どの類型がどの程度いるか」に対応。**役職保有でキーマン(A)を絞る**設計：

- **Tier A（重点）**: 政府三役（大臣・副大臣・政務官）または委員長・会長
- **Tier B（通常監視）**: 委員会所属あり（Aでない）
- **Tier C（様子見）**: 上記以外
- **seniority**（別属性）: 当選回数で senior/mid/junior。※既定では当選数を自動でAにしない
  （`senior_grants_tier_a: false`）。JCLPが重視するなら true に切替。

**現名簿713名での算出結果**（`compute_influence.py`）:

| Tier | 人数 | 衆 | 参 |
|------|------|----|----|
| A 重点 | 130 | 81 | 49 |
| B 通常監視 | 577 | 380 | 197 |
| C 様子見 | 6 | 4 | 2 |

seniority: senior 248 / mid 262 / junior 203。
→ これはSTEP3（発言分類）を待たずに得られる**エンゲージメント優先順位の土台**。

## 5. 分類の出力ルール（`output_rules`）

- 各スタンス判定に**根拠発言(quote＋出典)を必須**、確信度(0-1)を付す。
- **発言に無い立場は推測で埋めない** → `neutral_unknown`（真正性の担保）。
- 関連発言が `low_engagement_threshold`(=2) 未満なら `low_engagement` に寄せる（memo L65）。
- 賛成/反対の両面を客観的に扱う（memo L64）。

## 6. JCLPレビューで決めるべき論点（要確認）

1. **トピックの過不足**: dynamic に入れるべき論点（例: 系統・電気料金・カーボンプライシング）の追加是非。
2. **類型の定義**: L25の3類型の境界、`原子力・安定供給重視`等の追加類型の妥当性。
3. **スタンス5段階の粒度**: 5段階で足りるか（例: 「言及あるが立場不明」の扱い）。
4. **影響力の閾値**: `senior_grants_tier_a` を on にするか、当選回数の senior_min(=5)。
5. **関心度の判定基準**: 発言件数だけでなく主導性（提出・質問回数）を見るか。

## 7. パイロット検証の進め方（STEP3が通ったら）

メモ L58「一度観点・評価軸を設計すれば、同じ観点でデータを通す」を、いきなり全員でなく**5〜10名**で検証：

1. Tier A から数名＋発言の少なそうな数名を選ぶ（賛否・関心の幅が出るように）。
2. `fetch_speeches.py`（要・非遮断NW）で発言収集 → `classify.py` で①②③を分類。
3. 出力（類型＋根拠発言＋確信度）を武井氏がレビュー → `axes.yml` を微修正。
4. 精度が納得できたら全員へ展開（STEP3）。

※ 現在 STEP3 の会議録APIはこの環境で403遮断のため、パイロットは非遮断NW／プロキシで実施する。
